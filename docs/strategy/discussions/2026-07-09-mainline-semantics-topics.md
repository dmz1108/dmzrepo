### 1. 主线族边界怎么归并?

**问题:** 例如 `中芯/中芯国际`,以及 `光模块/CPO/交换机/算力AI`。

**Codex 观点:** 不建议永久压成一个平面名。应分为硬归一 alias、主线族 family、当日活跃粒度 active node。明显同义词硬合并;同族细分保留层级;当天谁有独立强度就展示谁。

**Claude 观点:** 分两层:同义重复(中芯/中芯国际)是数据质量问题,用可人工维护的别名词典硬归并,归并名用综合归纳里出现频次最高的规范名;族关系(光模块 vs 算力AI)不做静态合并,判定依据用"成分股重叠度 + 近30日主因共现"量化而非拍脑袋,两者当日各自有独立涨停/资金强度时允许并列成卡但标注亲缘关系。归并对错必须可复盘:预测复盘按主题记胜率,某归并规则导致胜率恶化即回滚,所有规则进配置文件可审计。

**互评 / 回应:**
- Claude 质疑 Codex: `active node` 没解决 parent/child 同族重复占位;半导体和存储芯片是否会同时占两张卡、重复计算强度?需要互斥规则。

### 2. 主线粒度切换怎么识别?

**问题:** 半导体突然下沉到存储芯片这类现象如何在盘中识别?

**Codex 观点:** 必须支持。用大族内部分化、细分股涨幅/冲板集中、细分原因集中、资金集中、领涨股一致性识别。当天只有存储芯片强,主线就应显示存储芯片,不是继续报半导体。

**Claude 观点:** 识别信号首选"细分原因集中度":当日大族内涨停/大涨股中,同一细分原因(取自涨停底库个股细分字段)占比达阈值(建议>=60%起讨论)、且大族其余成分股明显不涨(分化),即判定粒度下沉日,当日主线名切为细分。关键主张:历史证据按"股票集合"继承而非按"主题名"继承——存储芯片这批股此前涨停记的主因是半导体,但股票本身的涨停历史应作为存储芯片主线的证据,否则下沉首日会被误判成零证据的首日新题材。切回条件对称:集中度回落、大族多个细分同时扩散时回大族粒度。底层证据永远按股票+原文记录,粒度只是展示层决策,不改写历史。

**互评 / 回应:**
- Claude 质疑 Codex: 证据继承依赖"细分成分集合",但系统可能没有固定存储芯片成分表;细分级资金也暂不可得,不能写成判定条件。
- Codex 回应/修正: 接受。细分成分集合不能假设已有固定板块表,应由近30日个股细分原因命中同一细分词及别名形成,概念目录只做补充证据。细分级资金目前不能作为硬条件,只能作为未来有个股级资金后的加分项;现阶段识别依赖细分原因集中度、当日涨幅/冲板/涨停集中、parent 剩余成分是否不跟随。
- Codex 质疑 Claude: `>=60%` 适合做讨论起点,但不能单独硬编码;需要最小股票数和广度约束,否则 2 只股里 1 只命中也可能误判。建议规则是集中度 + 最小样本 + 相对 parent residual 强度三项共同成立。
- Claude 回应/修正: Pending。

### 3. 个股细分原因是否算主因证据?

**问题:** 细分证据能否单独让股票过 mainZt10 硬门槛?

**Codex 观点:** 应该算,但权重低于板块主因。建议拆成 boardEvidence 和 detailEvidence。细分原因可参与粒度下沉、扩池和解释;满足多股集中/今日上涨集中/非单股逻辑时可升级为主线证据。

**Claude 观点:** 应该算,理由直接来自议题2:如果细分原因某天能升级成当日主线,它平时就必须积累证据资格,否则下沉日全体龙头过不了 mainZt10 门槛,系统自相矛盾。具体口径:门槛(资格判定)上"板块主因或个股细分指向本主线"任一命中即算;打分权重上细分低于板块主因(细分更早但更碎、单股噪音多),且单股孤例细分不构成主线证据,需近10日>=2股或>=2次同细分才计入。

**互评 / 回应:**
- Claude 质疑 Codex: `boardEvidence/detailEvidence` 是打分概念,但硬门槛需要二值结论:细分证据能否单独让股票过 mainZt10 门槛?
- Codex 回应/修正: 明确表态:可以。若个股细分原因指向本 active node,且满足非孤例条件,应能单独通过主线关联门槛;否则粒度下沉日会没有龙头。打分仍分层:板块主因权重大于细分原因,多源/综合归纳确认大于单源细分。

### 4. 龙头的 10 日涨停次数口径

**问题:** 用总次数,还是本主线次数?

**Codex 观点:** 两个都要展示,但排序核心应看"本主线10日涨停"。全部10日涨停只能说明股性强,不能让股票靠别的题材涨停刷进当前主线龙头。

**Claude 观点:** 排名主口径应改为"与本主线相关的10日涨停次数"(板块主因或细分原因指向本主线,与议题3口径联动),总次数降级为展示和股性参考——现状总次数进排名确实有"靠别的题材刷榜"失真。但必须保留新鲜度保护:紫光股份案例说明主因会切换,最近一次涨停主因指向本主线的股票不能因更早涨停不相关而被压低,"最近一次主因即本主线"应给足权重。坚持公平打分原则:同口径下次数相等得分必须相等,不引入排名彩票或衰减系数。两个口径都展示给用户。

**互评 / 回应:**
- Claude 质疑 Codex: 本主线10日涨停次数与粒度下沉存在计算顺序循环;需先做证据继承再统计次数,并写入 owner 定过的公平原则。
- Codex 回应/修正: 接受。计算顺序应写死:先确定 family/active node 和细分股票集合;再做历史证据继承;再统计本主线10日涨停次数;最后排序。公平原则也应写入:同一口径下本主线10日涨停次数相等,次数基础分必须相等,不能用随机排名或隐含衰减破坏可解释性。

### 5. 一只股票能否同时属于多个主线?

**问题:** 唯一第一龙头的标准冲突、预测复盘重复计数怎么处理?

**Codex 观点:** 候选池允许多归属,展示时必须写清归属依据。若要选唯一第一龙头,再按今日活跃度、最近主因、资金/涨幅贡献归到最强 active node。

**Claude 观点:** 候选池允许多归属(现实就是交叉的,强行唯一会丢信息),这与 owner 定下的"一股可同时是龙头/明星/潜力"标签哲学一致。但"第一龙头"若需唯一,按最近一次涨停主因指向哪条主线归哪条。前端每张卡标注该股入选本卡的依据(哪次涨停、什么主因)。要注意预测复盘统计:同一股在两条主线都当龙头时,胜率按主线分别记录,不因一股两挂而重复计成两次独立成功。

### 6. 今日实时卡片数据怎么用到策略页?

**问题:** 实时卡复用的价值是提速还是分析?新鲜度语义怎么统一?

**Codex 观点:** 应作为盘中快速信号、交叉验证和扩池来源,不直接替代主因库。实时口径含当日盘中变化,收盘价库口径只到昨收,前端必须区分。

**Claude 观点:** 定位为交叉验证+扩池+当日口径补充,不替换主因库:实时卡名单可补主线龙头池的漏(07-08 星网锐捷被漏正是池不全的教训),实时强度可给主线升降权。两口径共存必须显式标注("含今日盘中" vs "截至昨收"),同一数值位置不混用,否则用户看到同股两个涨幅会怀疑数据错。最大限制是快照 15:30 才生成,盘中读到的是上一交易日卡片——纯盘中信号得靠主线榜自己的实时扫描,快照只能当历史背景;若要真复用,实时卡统计需支持盘中增量刷新,这是后续工程问题。扩池后主因硬门槛照旧适用,资格线不因数据来源放松。

## 议题清单

### 议题 1:主线族边界与主题归并

同义/包含关系的主题会出现重复主线或归属分歧。已观察到的案例:

- `中芯` 与 `中芯国际` 曾同时各成一条主线(canonicalTopicName 未归并)。
- `光模块`、`交换机`、`CPO` 与 `算力AI`:是同一主线族的细分,还是独立主线?

需要回答:归并规则是什么(词典、包含关系、成分股重叠度,还是别的);归并后主线名用哪个;归并错了怎么发现和回滚。

### 议题 2:主线粒度切换现象(owner 案例)

Owner 提供的现象,作为第一手证据记录:

> 有时候涨停主因一直涨的是半导体,但突然有一天涨停主因变成了涨停复盘里的个股细分原因:存储芯片。这种也是有可能的。所以突然某一天你会发现不是所有半导体都在涨,发现都是之前个股细分的存储芯片的股在涨,所以这一天今日主线就是存储芯片,会有这个现象。

也就是说:主线的正确粒度不是固定的。多数日子"半导体"是对的粒度,但某一天资金只做其中的"存储芯片"细分,那天的今日主线就应该是存储芯片,而不是继续报"半导体"。

需要回答:系统怎么在盘中识别"粒度下沉日"(比如大族内只有某细分的股在涨/涨停,其余成分股不动);识别信号用什么数据(细分原因分布、板块内涨跌分化、涨停股的细分原因集中度);下沉那天历史主因证据怎么继承(存储芯片股此前的涨停主因记的是"半导体",算不算存储芯片主线的历史证据);反过来,细分退潮回到大族时怎么切回来。

### 议题 3:个股细分原因是否算主因门槛证据

现状:龙头硬门槛 mainZt10Count 只认综合归纳的 finalBoardTopic(板块主因),个股细分原因不参与。

需要回答:个股细分原因要不要算入主因门槛/主因次数;算的话权重与板块主因是否等同;和议题 2 直接相关——如果细分原因可以某天升级为主线,那它平时是否也应该积累"证据资格"。

### 议题 4:龙头打分中 10 日涨停次数的口径

现状:三榜排名里的 `10日涨停次数` 用的是 zt10Count(该股全部涨停,不分原因),主因关联只体现在 mainZt10Count 硬门槛和新鲜度加分。

需要回答:排名榜该用总次数还是本主线主因次数;混用会不会出现"靠别的题材涨停刷榜"的失真;只用主因次数会不会漏掉刚切换主因的真龙头(参考案例:紫光股份最近一次涨停主因是算力,更早涨停主因不是)。

### 议题 5:一股双主线

同一只股票可能同时出现在两条主线的龙头池(例如既是算力AI又是CPO成员)。

需要回答:允不允许一股同时当两条主线的龙头;不允许的话按什么规则归一(最近一次主因?主因次数多的一边?);允许的话前端怎么展示才不误导。

### 议题 6:复用今日实时卡片成分股统计分析龙头(owner 思路)

Owner 思路:今日实时卡片已算好每板块成分股 10日/30日涨幅、10日涨停统计,有时可以直接用这些现成数据,结合复盘综合归纳的主因和个股细分原因,一起分析龙头。这正是"实时、复盘、策略三者相结合"。

已知的口径差异(事实,非观点):

- 实时卡 gain10/gain30 是 K 线算的,含当日盘中涨幅;主线榜龙头的 gain10/gain30 是收盘价库算的,盘中只到昨收。
- 实时卡快照 15:30 生成,盘中读到的是最近快照日的卡片数据;`computeBoardQiLeaders`(QI 龙头)已在用这份快照。
- 实时卡的成分股名单来自板块目录/涨停记录聚合,主线榜龙头池来自涨盘扫描+主因库补全,两个池子不完全重合。

需要回答:实时卡数据在龙头分析里扮演什么角色(替换指标口径?交叉验证?扩池?);盘中含当日涨幅的口径和收盘口径怎么共存,会不会造成同一股两处显示不同涨幅;快照滞后(盘中用的是上一交易日卡片)对盘中预测的影响;如果扩池,主因硬门槛是否照旧适用。

## Codex Independent View

### 总体判断

今日主线榜不应该是盘后复盘的复述,也不应该为了展示数量而勉强生成很多主线。它的定位应是:盘中用今日实时强势板块、资金、成分股涨幅/涨停冲击、近 30 日四源主因库,推断"今天资金最可能正在围绕什么方向形成合力"。如果真实主线只有 1-2 条,就只显示 1-2 条;若没有足够证据,宁可显示"正在形成/证据不足",不强行填满。

我建议把主题系统设计成"主线族 family + 当日活跃粒度 active node"两层:大主题负责继承历史语境,细分主题负责捕捉当天资金真正攻击的位置。例如"半导体"是 family,但当日只有存储芯片股集中上涨时,active node 应切到"存储芯片"。

### 议题 1:主线族边界与主题归并

不建议把所有相近词永久合并成一个平面名称。应分三类处理:

1. **硬归一 alias**:明确同义词必须合并,例如 `中芯` 与 `中芯国际`。这类由词典维护,可人工修正。
2. **主线族 family**:例如 `算力AI` 下可包含 `光模块`、`CPO`、`交换机`、`液冷` 等。它们不是永远合并,而是同族不同粒度。
3. **当日独立主线 active node**:如果某细分方向当天有独立强度、资金、涨停冲击和成分股集中性,它可以从 family 里单独升出来展示。

归并规则不能只靠文字包含。建议综合:

- 主题别名词典:解决 `中芯/中芯国际` 这类明显重复。
- 成分股重叠度:重叠高但名称不同,提示可能同族或重复。
- 当日上涨/涨停股重叠度:判断当天是否其实在做同一批股票。
- 主因库共现:近 30 日同一批个股是否经常在两个主题之间切换。
- 资金与涨幅集中:决定当天展示大主题还是细分主题。

回滚方式:所有 alias/family 规则都应进入可审计配置或调试接口,前端显示"归并依据";如果发现归并错,可以把该规则禁用,并重新生成当天主线榜。

### 议题 2:主线粒度切换现象

Owner 的"半导体 -> 存储芯片"案例是今日主线榜必须支持的核心能力。我的判断是:系统应允许同一 family 在不同日期显示不同粒度。

盘中识别"粒度下沉日"可用以下信号:

- 大主题上涨不均匀:半导体整体不都涨,但存储芯片子集明显集中上涨。
- 子集强度更高:子集内涨幅中位数、涨停/冲板数量、涨幅大于阈值股票占比,显著高于 family 其余股票。
- 细分原因集中:当天上涨/涨停股的历史个股细分原因大量指向 `存储芯片` 或其别名。
- 资金集中:如果可拿到资金,子集净流入或成交放大明显优于 parent residual。
- 领涨股一致:子集里出现新的龙头候选,而 parent 里其他方向没有同步跟随。

历史证据继承不能简单全继承。建议:

- 如果历史主因是 `半导体`,且该股属于 `存储芯片`成分/细分原因曾出现存储芯片,可给 `存储芯片` 低权重继承证据。
- 如果历史主因明确是 `存储芯片`,给满权重。
- 如果只是同属半导体但没有存储芯片证据,不能给存储芯片满权重。

切回 parent 的条件:细分集中度下降,多个半导体细分同时扩散,或者 parent 的广度/资金/涨停冲击重新强于任何单一 child。

### 议题 3:个股细分原因是否算主因门槛证据

个股细分原因应该算证据,但不应和板块主因完全等同。原因是:今日主线榜本质上要预测当日资金攻击点,个股细分原因往往比板块主因更早、更细,但它也更碎、更容易噪音。

建议拆成两个指标:

- `boardEvidence`:综合归纳板块主因证据,权重高,稳定。
- `detailEvidence`:个股细分原因证据,权重中等,用于粒度下沉、扩池和解释。

当细分原因满足这些条件时,可以升级为主线证据:

- 多只股票同日/近几日反复出现同一细分原因。
- 细分原因和今日实时上涨股票高度集中。
- 至少部分来源或综合归纳能支持这个细分不是孤立备注。
- 细分方向有足够成分股,不是单股逻辑。

因此,主因门槛不应只看 `finalBoardTopic`;应允许 `finalBoardTopic + finalStockReason/detailReason` 共同形成"主线证据",但权重分层。

### 议题 4:龙头打分中 10 日涨停次数的口径

只用总涨停次数会有失真:一只股票可能靠别的题材涨停很多次,然后被错误放进当前主线龙头。只用本主线次数又会漏掉主线切换早期的真正强股。

建议前端和打分同时保留两个口径:

- `本主线10日涨停`:近 10 日涨停中,主因/细分原因能指向本主线的次数。作为主排名核心。
- `全部10日涨停`:该股所有涨停次数。作为强度补充和展示字段。

龙头排序建议:

1. 先过主线关联门槛:至少有板块主因或细分原因证据,不能纯靠总涨停次数入榜。
2. 排名主要看本主线涨停次数、最近一次主线相关涨停的新鲜度、今日盘中涨幅/冲板状态。
3. 10日/30日涨幅用于识别趋势龙头和潜力股,但不能替代主因关联。
4. 全部涨停次数只作为辅助,用于说明股性强,不直接决定主线归属。

### 议题 5:一股双主线

允许一只股票同时出现在多个主线的候选池,但前端必须避免误导。现实中很多股票确实同时属于 `算力AI`、`CPO`、`光模块` 等多个方向。

建议:

- 在候选池层允许多归属。
- 在每张主线卡里显示"本卡归属依据",例如最近主因、细分原因、所属成分股。
- 如果同一只股在多张卡都进入龙头候选,显示辅助标签:如"同时属于 CPO/算力AI"。
- 如果需要选"今日第一龙头",再做唯一归属:优先归给今日盘中更活跃、主因证据更近、资金/涨幅贡献更大的 active node。

这样既不丢真实交叉属性,也不会让用户误以为同一只股票在每个方向里的逻辑都一样强。

### 议题 6:复用今日实时卡片成分股统计分析龙头

我同意复用今日实时卡片里的成分股统计,但它的角色应该是"盘中快速信号 + 交叉验证 + 扩池",不是直接替换主因库。

使用方式:

- **提速**:今日实时卡片已经算过成分股 10日/30日涨幅、10日涨停统计,策略页可以优先读取这些结果,减少重复计算。
- **扩池**:如果主因库里某主题证据强,但策略候选池漏了实时卡片里正在大涨的核心成分股,应补进候选池再检查主因/细分原因。
- **交叉验证**:某主线只有历史主因但今日实时卡片没有涨幅/资金/成分股响应,应降权;反过来,实时强但主因库弱,可以标记为"新发方向/待确认",不应直接当成熟主线。

口径必须明确:

- 实时卡片的 10日/30日涨幅含当日盘中变化,适合盘中预测。
- 收盘价库的 10日/30日涨幅只到昨收,适合稳定复盘和历史对照。
- 前端不要把两种口径混在同一个字段里。建议显示为"盘中涨幅/盘中趋势"与"收盘口径趋势"。

关于快照滞后:如果今日实时卡片盘中读到的是上一交易日快照,那它只能做历史背景,不能当今日实时证据。今日主线榜在开盘前不应显示昨日主线来冒充今天;交易开始后一段时间内如果新证据不足,可以显示"昨日快照参考/今日未形成",而不是生成假主线。

### 建议的收敛顺序

我建议先收敛 1+2+3,因为这决定"主线到底是什么";再收敛 4+5,因为这决定"谁是龙头";最后收敛 6,把实时卡片数据接进策略系统做加速和交叉验证。

如果要先落地一个低风险版本,我建议:

1. 增加主题 family/active node 的解释字段,不急着大规模改 UI。
2. 龙头候选同时展示"本主线10日涨停"和"全部10日涨停"。
3. 细分原因先参与加权和解释,暂不直接完全替代板块主因。
4. 实时卡片数据先做候选补充与降权/升权信号,不直接一票决定主线。

### Claude Challenges Codex

先说认同的:family + active node 两层结构、细分原因分层计权、双口径并行展示、收敛顺序 1+2+3 → 4+5 → 6,这些我都同意,不重复论证。以下是我认为 Codex 观点里不完整、难落地或必须说得更死的地方:

**质疑 1(议题 1/总体):active node 的展示规则没有解决"同族重复占位"。**
"当天谁有独立强度就展示谁"听起来对,但没回答 parent 和 child 同时强时怎么办:半导体和存储芯片会不会同时各占一张主线卡?同一批股票的涨停/资金强度会被计进两张卡,predictScore 排名里一个族占掉榜单多个头部位置,主线数量和强度都被重复计算夸大。需要明确的互斥规则,我的建议起点:同族默认只出一个粒度的卡(下沉日出 child,平常出 parent),仅当两个 child 之间成分重叠低且各自独立强时才允许同族并列;族内股票的强度贡献不允许计入两张卡。这条不定死,议题 1 的归并讨论等于没完成。

**质疑 2(议题 2):证据继承的分级依赖一个不存在的名单。**
Codex 的三档继承(存储成分/曾出现存储细分→低权;明确存储主因→满权;仅同属半导体→不满权)前提是系统知道"谁是存储芯片成分股"。但存储芯片不是板块目录里的固定板块,涨停底库的个股细分原因文本是唯一来源。所以分级规则必须先定义"细分成分集合怎么产生":我的主张是集合就由历史细分原因命中来定义(近 N 日细分原因含"存储芯片"及其别名的股票集合),东财/同花顺概念目录只做补充。另外信号清单里的"资金集中"要标注现状不可得:资金净流入是板块级的,细分子集级的资金数据现在拿不到,盘中识别只能靠涨幅/涨停/细分集中度三项,别把不可得的信号写进判定条件。

**质疑 3(议题 3):"至少部分来源或综合归纳能支持"太模糊,且门槛问题没回答死。**
四源共识已经有现成的 consensusTier 机制,升级条件应直接引用它(例如细分原因至少来自两源或综合归纳确认),不要再造一个模糊的"部分来源支持"。更重要的是:boardEvidence/detailEvidence 分层计权说的是打分,但现行 mainZt10≥1 是硬门槛(布尔),Codex 没有明确回答"细分证据能不能单独让一只股过门槛"。我的立场是能(否则议题 2 下沉日无人过门槛),请 Codex 明确表态是/否,这必须是二值结论,不能停在"权重分层"。

**质疑 4(议题 4):"本主线10日涨停"和议题 2 存在计算顺序上的循环依赖,且公平原则没有写进去。**
下沉日"本主线"=存储芯片,而这批股的历史涨停主因记的都是"半导体"——如果先按主题名统计本主线次数再排龙头,下沉日全体候选的本主线次数为 0,排名退化。必须先做议题 2 的证据继承(按股票集合),再在继承后的口径上统计"本主线次数",计算顺序要写进决议。另外 owner 定过的公平原则(次数相等得分必须相等)在 Codex 的排序方案里没有出现,任何"新鲜度+今日盘中+次数"的组合打分都必须显式保证这一条,请补进修正稿。

**质疑 5(议题 5):唯一第一龙头的三个标准会打架。**
"今日盘中更活跃、主因证据更近、资金/涨幅贡献更大"是三个可能互相矛盾的维度(A 股今日更活跃、B 股主因更近很常见),没有给出优先序或合成公式,落地时每个人实现出来都不一样。要么定字典序(建议:最近主因指向 > 今日活跃 > 贡献),要么明确说复用现有综合分不另造公式。另外补一个 Codex 没提的点:一股两挂时预测复盘的胜率统计要按主线分别记,不能一次涨停在两条主线里重复计成两次预测成功,这直接影响回测数据的可信度。

**质疑 6(议题 6):"提速"这个收益基本不成立,别让它进决议依据。**
主线榜龙头指标读的是本地收盘价库+涨停底库,PR #10 之后还有 60 秒缓存,贵的从来不是这段;真正贵的 K 线拉取恰恰发生在实时卡片自己身上(每板块 60 股×并发 6)。所以"策略页优先读实时卡结果减少重复计算"的提速收益接近零,实时卡数据的价值只在分析层(扩池/交叉验证/当日口径),决议里应把"提速"从理由里拿掉,免得为一个不存在的收益引入快照滞后的复杂度。另外"开盘后新证据不足显示昨日快照参考"要和已收敛的 Step A 契约对齐措辞:Step A 已规定盘前不得把昨日数据冒充今日,这里的"昨日快照参考"必须复用 Step A 的 staleness/snapshot 标注体系,不要新增第二套新鲜度语义。
